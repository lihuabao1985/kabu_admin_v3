# 执行建议（让 Codex 跑得最稳）

## 你每次执行一个 Schema 文件，用固定三步：

- 只读分析
“阅读 10_SCHEMA_XXX.md，总结要改哪些文件，给计划，不改代码。”
- 执行
“按计划执行，只改允许目录，必须更新 OpenAPI，必须跑 mvn clean verify。”
- 验收
“列出 diff、测试结果、风险、回滚。”

五、Codex CLI 的“对话套路”：三步把成功率拉满
第 1 步：侦察（只读）

给它一句：

“阅读仓库结构，列出后端/前端入口、构建命令、测试命令、MyBatis mapper 位置，输出一页总结，不改文件。”

第 2 步：计划（先给方案再动手）

“先给改动计划（涉及哪些文件、怎么改、测试怎么补），我确认后再执行。”

（你不想确认也行，但这一步能大幅减少它乱改。）

第 3 步：执行（按护栏施工）

“按计划修改，仅改允许目录。完成后跑测试并给我 diff + 命令记录。”

六、给你一套“专用提示词”——专治 MyBatis 容易写崩

每次涉及 MyBatis，追加这段附加要求，效果很明显：

SQL 必须可读：字段显式列出，禁止 SELECT *

动态条件必须用 <where> + <if>，避免手写 1=1

IN 条件用 <foreach>，并处理空 list（空则不拼或直接返回空结果）

参数命名与 Java DTO 字段一致

复杂 SQL 必须补注释，解释业务口径
